# /tech-review — 技术架构评审

向用户展示技术设计以获取人工批准。这是强制关卡 — 未经批准不得进入实现阶段。

## 流程

### 第一步：定位设计

```bash
pnpm exec openspec status --change <变更名> --json
```

读取技术设计：`openspec/changes/<变更名>/design.md`

### 第二步：展示评审

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  技术架构评审
  变更：<变更名>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 技术栈
- 前端：[框架 + 语言]
- 后端：[框架 + 语言]
- 数据库：[数据库 + ORM]

## 架构概览
[高层描述]

## 数据模型
[核心实体和关系]

## API 端点
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/v1/... | ... |
| GET  | /api/v1/... | ... |

## 关键取舍
| 决策 | 原因 | 备选方案 |
|------|------|----------|
| ... | ... | ... |

## 风险点
- [风险 1]
- [风险 2]

完整设计：openspec/changes/<变更名>/design.md
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 第三步：收集决定

"请审阅以上技术架构：
- **批准** — 架构合理，进入实现阶段
- **修改：[反馈]** — 我会根据反馈更新
- **讨论** — 先讨论具体技术决策
- **拒绝** — 重新设计架构

你的决定是？"

### 第四步：处理决定

- **批准**：在 design.md 中插入 `<!-- FORGE_GATE:TECH_REVIEW:APPROVED:YYYY-MM-DD -->`（用实际日期）
- **修改**：更新设计 → 重新展示（最多 5 轮）
- **讨论**：技术讨论 → 更新设计 → 重新展示
- **拒绝**：记录原因，通知调用方重新设计
