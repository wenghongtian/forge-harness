# 后端开发规则

## API 设计
- 默认 RESTful，除非技术栈指定 GraphQL
- 面向资源的 URL：`/api/v1/[资源]`
- 使用正确的 HTTP 方法：GET（读取）、POST（创建）、PUT（全量更新）、PATCH（部分更新）、DELETE（删除）
- 分页：大数据集用游标分页，小数据集用偏移分页
- 过滤：查询参数使用清晰命名（`?status=active&sort=-created_at`）
- 版本管理：URL 前缀 `/api/v1/`

## 请求/响应规范

### 成功响应
```json
{
  "data": { ... },
  "meta": {
    "page": 1,
    "total": 100,
    "hasMore": true
  }
}
```

### 错误响应
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "人类可读的描述",
    "details": [
      { "field": "email", "message": "邮箱格式无效" }
    ]
  }
}
```

## 输入验证
- 在控制器/处理器层验证所有输入
- 使用模式验证（Zod、Joi、Pydantic 等）
- 永远不要只依赖客户端验证
- 清理字符串以防止 XSS
- 参数化所有数据库查询以防止 SQL 注入

## 业务逻辑
- 业务逻辑放在 `services/`，不放在控制器中
- 控制器：解析请求 → 调用服务 → 格式化响应
- 服务：纯业务逻辑，不涉及 HTTP 概念
- 模型：数据访问和 ORM 操作
- 保持控制器精简，服务可测试

## 数据库
- 始终使用迁移进行 schema 变更
- 编写幂等迁移（可安全重复运行）
- 为外键和高频查询列建索引
- 多表操作使用事务
- 合理配置连接池

## 认证
- 认证中间件作为链中的第一个中间件
- 会话使用 httpOnly、secure、sameSite 的 Cookie（或 JWT 用 Authorization 头）
- 长时间会话的令牌刷新机制
- 认证端点（登录、注册、重置密码）添加速率限制
- 多次登录失败后锁定账户

## 日志与监控
- 结构化 JSON 日志（不用 console.log）
- 日志级别：error、warn、info、debug
- 所有日志包含请求 ID 用于链路追踪
- 记录所有入站请求（方法、路径、状态、耗时）
- 永远不要记录敏感数据（密码、令牌、个人信息）

## 错误处理
- 全局错误处理器捕获所有未处理错误
- 生产环境响应中不暴露堆栈信息
- 内部错误映射为合适的 HTTP 状态码
- 服务端记录完整错误细节，客户端返回安全消息

## 测试
- 服务层单元测试（模拟依赖）
- API 端点集成测试（优先使用真实数据库）
- 测试错误场景，不只是正常路径
- 使用工厂/Fixture 生成测试数据
- 每次运行间清理测试数据
