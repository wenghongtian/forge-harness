# 产品设计方法论

## 何时使用
当需要从用户需求出发设计产品规格时引用此技能。适用于 `/genesis` 和 `/iterate` 指令的产品设计阶段。

## 需求分析框架

### 问题定义（5W1H）
1. **Who** — 谁遇到了这个问题？（用户画像）
2. **What** — 具体的痛点是什么？（不是"想要什么功能"）
3. **When** — 什么场景下遇到？（触发条件）
4. **Where** — 在什么环境中？（设备、网络、场所）
5. **Why** — 为什么现有方案不够？（竞品分析）
6. **How** — 用户现在是怎么绕过的？（变通方案暗示真正需求）

### 用户故事编写规范
```
作为 [用户角色]，
我想要 [完成某个操作]，
以便 [获得某个价值/结果]。
```

**好的用户故事**：
- 作为新注册用户，我想要通过邮箱快速注册，以便在 30 秒内开始使用产品。

**差的用户故事**：
- 作为用户，我想要一个注册页面。（缺少价值，缺少约束）

### 验收标准编写规范
使用 GIVEN/WHEN/THEN 格式：
```
假设 [前置条件]
当 [用户操作]
则 [系统响应]
```

每个标准必须是**可测试的**：
- 好：「输入超过 50 个字符时，显示红色错误提示"最多 50 个字符"」
- 差：「输入过长时给出提示」（多长？什么提示？）

## 功能优先级判断

### MoSCoW 方法
- **Must have** — 没有就无法上线的核心功能
- **Should have** — 重要但有替代方案的功能
- **Could have** — 锦上添花，时间允许就做
- **Won't have** — 明确排除，写入范围外

### 80/20 原则
- 80% 的用户价值来自 20% 的功能
- 先识别这 20%，做到极致
- 剩余的功能可以在后续 `/iterate` 中逐步添加

## 信息架构设计

### 页面层次
1. **一级页面** — 核心功能入口（导航直达）
2. **二级页面** — 功能详情（从一级页面进入）
3. **模态/抽屉** — 轻量操作（不离开当前页面）

### 导航原则
- 主导航项不超过 7 个（7±2 法则）
- 每个页面有且只有一个主要操作（CTA）
- 用户在 3 次点击内能到达任何功能
